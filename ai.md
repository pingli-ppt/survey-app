# 项目报告

---

## 一、项目目标

本项目实现了一个基于 MongoDB 的在线问卷系统。系统整体参考常见问卷工具的基本功能，但在实现上做了简化，重点放在数据结构设计和核心逻辑上。

系统支持用户注册登录、创建问卷、设计题目、填写问卷以及查看统计结果。相比简单的表单系统，本项目额外实现了题目跳转逻辑和较完整的数据校验机制。

在开发过程中，主要目标不是做复杂界面，而是把数据流和逻辑跑通，并保证结构具有一定扩展性。

在第二阶段，新增了题目保存与复用、题目共享、题目版本管理、题库管理、题目使用追溯与统计的功能

---

## 二、系统整体设计

系统采用典型的前后端分离结构：

* 前端：原生 HTML + JavaScript，实现交互和页面渲染
* 后端：Node.js + Express 提供 API
* 数据库：MongoDB（通过 Mongoose 管理）

整体流程：

> 用户登录 → 从题库选择或新建题目创建问卷 → 添加跳转逻辑 → 发布 → 他人填写 → 系统统计结果

---

### 模块划分

系统可以拆成几个比较清晰的部分：

#### 1. 用户模块

负责注册、登录以及身份校验（JWT）

#### 2. 问卷模块

负责问卷创建、发布、关闭、查询等操作，问卷题目直接关联到题库中题目id（不同版本题目id不同）

#### 3. 题目库与版本管理模块

题目公共库管理（每道题目都进入个人题库，可以设置状态是私有还是公开从而实现共享），题目多版本控制，题目使用情况统计

#### 4. 答卷模块

保存用户填写的答案，同时记录设备信息（IP、UA）

#### 5. 统计模块

根据答卷数据动态计算结果

## 三、MongoDB 数据库设计

---

### （1）第一阶段：初始数据库设计
#### 第一阶段集合（3 个）
1. **User（用户）**
2. **Survey（问卷 + 题目嵌套）**
3. **Response（答卷）**

#### 第一阶段结构特点
- **题目直接嵌套在 Survey 中**
- 题目不可复用
- 无版本、无题库、无共享
- 结构简单，适合最小可用版本
---
### （2）新需求带来的问题
当系统需要支持**题库、版本、复用、共享、安全编辑**时，旧设计出现严重问题：

#### 1. 题目无法复用
- 每个问卷独立存题目
- 相同题目重复创建
- 无法统一管理

#### 2. 修改题目会污染旧数据
- 编辑题目 → 直接覆盖原内容
- 已发布问卷的题目被改变
- **历史答卷与题目不匹配**

#### 3. 无法追踪题目使用情况
- 不知道题目被哪些问卷使用
- 不知道有多少人回答
- 无法安全编辑

#### 4. 没有版本管理
- 题目改坏无法回滚
- 无法查看历史
- 不支持“安全修改”

#### 5. 不支持共享/公开题库
- 题目只能自己用
- 无法设置公开/私有
- 无法指定共享用户

---

### （3）数据库结构修改了哪些部分？
#### 核心改动：题目从 Survey 内嵌 → 独立为题库多版本体系

#### 新增 3 个核心集合：
1. **QuestionBank（题目基础库）**
2. **QuestionVersion（题目版本）**
3. **QuestionUsage（题目使用记录）**

#### 修改原有集合：
1. **Survey**
   - 不再存题目内容
   - 只存**题目引用（baseId + versionId）**
2. **Response**
   - 保留不变
3. **User**
   - 保留不变

---

#### 1. User（用户）

```js
{
```

```js
  createdAt,       // 创建时间
  updatedAt        // 更新时间
}
```

#### 3. QuestionVersion（题目版本）
```js
{
  versionId,       // 版本唯一ID
  baseId,          // 所属题目基础库ID
  version,         // 版本号 1/2/3...
  parentVersionId, // 父版本ID（回滚/衍生用）
  title,           // 题目标题
  type,            // 题型 single_choice/multi_choice/text/number
  config,          // 题目配置（选项、范围、长度等，完整示例见下方）

}
```

#### 补充：config 字段完整示例结构（按题型区分）
```js
// 1. 单选题（single_choice）
config: {
  options: [       // 选项列表
    { value: "A", label: "非常满意" },
    { value: "B", label: "满意" },
    { value: "C", label: "一般" },
    { value: "D", label: "不满意" }
  ]

// 2. 多选题（multi_choice）
config: {
  options: [       // 选项列表
    { value: "1", label: "篮球" },
    { value: "2", label: "足球" },
    { value: "3", label: "羽毛球" }

  minSelect: 1,    // 最小选择数
  maxSelect: 2     // 最大选择数

// 3. 文本题（text）

// 4. 数字题（number）

```

#### 4. QuestionUsage（题目使用记录）
```js
{
  versionId,       // 题目版本ID
  baseId,          // 题目基础ID
  surveyId,        // 使用该题目的问卷ID
  responseCount,   // 该版本在该问卷中的回答次数
  lastUsedAt,      // 最后使用时间
  createdAt        // 记录创建时间
}
```

#### 5. Survey（问卷）
```js
{
  surveyId,         // 问卷公开ID
  title,            // 标题
  description,      // 描述
  allowAnonymous,   // 是否允许匿名
  allowMultipleSubmit, // 是否允许多次提交
  status,           // draft/published/closed
  deadline,         // 截止时间
  creatorId,        // 创建者ID
  questions: [      // 题目引用列表（只存关联，不存内容）
    {
      questionBankId, // 题库ID
      versionId,      // 固定版本ID
      order,          // 显示顺序
      logic           // 本问卷内跳转规则（完整示例见下方）
    }
  ],
  createdAt,
  updatedAt
}
```

#### 补充：logic 字段完整示例结构
```js
logic: {
  rules: [         // 跳转规则列表（按priority排序）
    {
      conditions: [ // 触发条件（可多个，同时满足）
        {
          type: "option_selected", // 条件类型：单选选中某选项
          optionValue: "A"         // 选中的选项值
        }
      ],
      targetQuestionId: "QB_123456", // 跳转目标题目baseId
      priority: 1                   // 优先级（数字越小，越先执行）
    },
    {
      conditions: [
        {
          type: "value_range",      // 条件类型：数字在某范围
          min: 18,                  // 最小值
          max: 30                   // 最大值
        }
      ],
      targetQuestionId: "QB_654321",
      priority: 2
    }
  ]
}
```
#### 6. Response（答卷）
```js
{
  surveyId,         // 所属问卷ID
  responseId,       // 答卷唯一编号
  respondentId,     // 填写用户ID
  respondentName,   // 填写人姓名
  isAnonymous,      // 是否匿名
  answers: [        // 答案列表
    { questionId, value }
  ],
  ipAddress,        // 填写IP
  userAgent,        // 设备/浏览器信息
  createdAt         // 提交时间
}
```

## 四、为什么这样设计

### 系统中的冗余设计与原因

#### （1）User 表中冗余 `survey_ids` 数组
```js
survey_ids: [ObjectId]  // 保存该用户创建的所有问卷ID
```

**为什么冗余？**
- **快速获取“我的问卷列表”**
  不用跨表关联 `Survey.creatorId` 再查询

---

#### （2）QuestionBank 中冗余 `usageCount` 使用次数
```js
usageCount: Number
```

**为什么冗余？**
- 不用每次都去 `QuestionUsage` 中 `countDocuments()`
- 题库列表页**加载极快**，避免大量计数查询拖慢系统

---

#### （3）QuestionUsage 冗余 `responseCount` 回答数量
```js
responseCount: Number
```

**为什么冗余？**
- 不用每次去 `Response` 中统计表数量，统计页面性能大幅提升

---

#### （4）Survey 中冗余题目固定版本号 versionId
```js
questions: [{
  questionBankId: String,
  versionId: String    // 冗余版本ID
}]
```

**为什么冗余？**
- 题目修改后，**旧问卷不受影响**

---

### 冗余设计总结
本系统采用**空间换性能**的冗余设计，包括用户问卷列表、题目使用次数、问卷固定版本号等。冗余设计的目的是**减少关联查询、提升页面加载速度、保证历史数据安全、提升系统整体稳定性**，在问卷系统这类查询频繁、结构多变、层级较深的场景中具有显著优势。

### 为什么不使用关系型数据库
问卷系统具有**层级结构深、字段不固定、配置灵活、逻辑嵌套**等特点。若使用 MySQL 等关系型数据库，需要拆分为多张数据表，使用大量 JOIN 操作，结构复杂、维护困难、扩展性差。

### 为什么适合 MongoDB
MongoDB 是文档型数据库，天然支持嵌套对象、数组、灵活结构，与问卷系统的数据结构高度匹配。其无范式约束的特性便于冗余优化，单文档查询即可获取完整数据，无需联表，大幅提升开发效率与系统性能，非常适合问卷、题库、版本化这类业务场景。

## 五、核心逻辑设计


### 1. 校验逻辑：整体流程没变，来源改动
#### 1. 没变的部分
所有题目统一通过一个函数处理：

```js
validateAnswer(question, answer)
```

不同题型在 switch 中处理：

* 单选：是否合法选项
* 多选：数量限制
* 文本：长度限制
* 数字：范围 + 是否整数

优点：

* 所有校验集中在一处
* 易维护

#### 2. 改动的地方
- **题目配置不再来自问卷，而是来自【题目版本】**
- 校验的**规则来源从问卷内嵌 → 题库版本化读取**
- 同一题目被多个问卷引用，**校验规则保持一致**
- 编辑题目后，**只有新版本会使用新校验规则**
  旧版本问卷依然使用旧规则，不会被影响

---

### 2. 跳转逻辑：结构基本不变，绑定关系改变
#### 1. 没变的部分
核心函数：

```js
getNextQuestionId()
```

执行流程：

1. 找到当前题目
2. 遍历 logic（按 priority 排序）
3. 判断 conditions 是否满足
4. 返回目标题目 ID

设计思路：

* 跳转规则写在数据库
* 后端只负责解释规则

这样做的好处：

* 不需要改代码
* 可以随意配置跳转

#### 2. 改动的地方
- **跳转逻辑不再跟着“题目内容”走，而是跟着“问卷里的引用”走**
- 逻辑规则 **存在问卷内部（Survey.questions.logic）**，不属于题库
- 题库只负责题目内容，问卷负责题目顺序与跳转，职责更清晰

## 关键接口详解

### 1. 添加题目
`POST /api/add-question`
- 请求参数说明：
  - `type` 可选值：`single_choice`, `multi_choice`, `text`, `number`
  - `config` 根据类型不同包含不同字段：
    - 单选/多选：`options`（数组，每项含 `value`, `label`）
    - 多选额外：`minSelect`, `maxSelect`
    - 文本：`minLength`, `maxLength`
    - 数字：`minValue`, `maxValue`, `integerOnly`

---

### 2. 添加跳转逻辑（问卷内题目）
`POST /api/add-logic`
- 支持的 `conditions` 类型：
  - `option_selected`：单选选中某个值
  - `option_any`：多选答案中包含某个值
  - `value_range`：数字在某个范围内（含边界）
- 参数结构：
  ```json
  {
    "surveyId": "...",
    "sourceQuestionId": "...",
    "logic": {
      "rules": [
        {
          "conditions": [{ "type": "...", "optionValue": "...", "min": "...", "max": "..." }],
          "targetQuestionId": "...",
          "priority": 1
        }
      ]
    }
  }
  ```

---

### 3. 创建题库题目
`POST /api/questions/create`
- 请求参数说明：
  - `type` 同上
  - `config` 同上
  - `isPublic`：是否公开，公开后其他用户可使用
  - `changeNote`：修改说明（用于版本记录）
- 返回结构：
  ```json
  {
    "success": true,
    "question": { "baseId": "...", "currentVersion": 1 }
  }
  ```
---

### 4. 向问卷添加题库题目
`POST /api/survey-question/add`
- 功能：把题库题目（指定版本）添加到问卷中
- 请求参数：
  ```json
  {
    "surveyId": "...",
    "questionBankId": "...",
    "versionId": "...",
    "order": 0,
    "logic": {}
  }
  ```
- 特点：
  - 绑定固定 `versionId`，保证问卷使用的题目版本永远不变
  - 可单独设置该题在本问卷中的跳转逻辑

---

### 5. 获取问卷统计
`GET /api/survey-stats/:surveyId`
- 响应结构：
  - 包含总答卷数 `totalResponses`
  - 每道题的统计结果，根据题型不同：
    - 单选：各选项计数与百分比
    - 多选：各选项计数、总选择次数、人均选择数
    - 文本：所有回答列表
    - 数字：求和、平均值、最小值、最大值

## 第十部分：遇到的问题与解决方案

---

### 问题1：创建问卷后所有题目变成undefined，无法添加跳转逻辑

**现象描述**：
- 创建问卷后进入编辑页面，题目列表显示为 undefined
- 无法为题目添加跳转逻辑

**排查过程**：
1. 检查API返回数据：发现问卷的题目数组有数据，但标题、题型等字段为空
2. 对比新旧数据结构：发现第二阶段题目不再内嵌在问卷中，改为引用方式（通过题库ID和版本ID关联）
3. 定位根本原因：前端代码仍按旧结构读取题目内容，未根据版本ID去查询题目版本表获取真实内容

**解决方案**：
- 修改问卷详情接口，在返回数据前先根据版本ID查询题目版本表
- 将查询到的题目标题、题型、配置等信息填充到返回数据中
- 确保前端接收到完整的题目信息

---

### 问题2：问卷内跳转异常，跳转至错误题目或跳转失效

**现象描述**：
- 设置跳转规则（如：选A → 跳第5题）后，实际跳转到其他题目
- 题目更新版本后，原有的跳转规则失效

**排查过程**：
1. 检查跳转规则配置：确认数据库中的目标题目ID正确
2. 检查题目版本：发现跳转逻辑只根据基础ID匹配，未考虑同一题目的不同版本
3. 定位根本原因：问卷引用的是具体版本ID，但跳转规则匹配的是基础ID，导致版本更新后匹配失败

**解决方案**：
- 将跳转逻辑改为基于问卷内的题目顺序（第几题）而非题目ID
- 规则中存储目标题目的顺序索引，跳转时根据顺序查找
- 添加优先级机制，多条规则按优先级顺序匹配

---

### 问题3：跨问卷统计数据不准确

**现象描述**：
- 同一题目在不同问卷中的回答无法汇总统计
- 题目更新版本后，新旧版本的回答混在一起统计

**排查过程**：
1. 检查统计接口：发现只统计单个问卷的回答数据，未跨问卷聚合
2. 检查数据关联：题目使用记录表记录了题目被哪些问卷使用，但统计时未使用这张表
3. 检查版本区分：统计时未按版本ID分组，导致不同版本数据混淆
4. 定位根本原因：统计逻辑未关联题目使用记录表，也未按版本区分数据

**解决方案**：
- 修改统计接口，先查询题目使用记录表获取所有使用该题目的问卷ID列表
- 再查询这些问卷的所有回答数据
- 按版本ID分组统计，区分不同版本的回答情况
- 对于数字题，计算平均值、最小值、最大值；对于选择题，统计各选项分布
